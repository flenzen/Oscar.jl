Unrestricted partitions

partitions â Method
partitions(n::IntegerUnion)

A list of all partitions of an integer $n â¥ 0$, produced in lexicographically descending order. This ordering is like in Sage, but opposite to GAP. You can apply the function reverse to reverse the order. As usual, you may increase performance by using smaller integer types. The algorithm used is "Algorithm ZS1" by Zoghbi & Stojmenovic (1998); see A. Zoghbi, I. Stojmenovic (1998).

Examples

julia> partitions(Int8(4))
5-element Vector{Partition{Int8}}:
 Int8[4]
 Int8[3, 1]
 Int8[2, 2]
 Int8[2, 1, 1]
 Int8[1, 1, 1, 1]

partitions(m::IntegerUnion, n::IntegerUnion, l1::IntegerUnion, l2::IntegerUnion; only_distinct_parts::Bool = false)

A list of all partitions of an integer $m â¥ 0$ into $n â¥ 0$ parts with lower bound $l1 â¥ 0$ and upper bound $l2 â¥ l1$ for the parts. There are two choices for the parameter only_distinct_parts:

  • false: no further restriction (default);
  • true: only distinct parts.

The partitions are produced in decreasing order.

The algorithm used is "parta" in W. Riha, K. R. James (1976), de-gotoed from old ALGOL code by E. Thiel!

source
partitions(mu::Vector{IntegerUnion}, m::IntegerUnion, v::Vector{IntegerUnion}, n::IntegerUnion)

All partitions of an integer $m >= 0$ into $n >= 1$ parts, where each part is an element in $v$ and each $v[i]$ occurs a maximum of $mu[i]$ times. The partitions are produced in decreasing order. The algorithm used is a de-gotoed version (by E. Thiel!) of algorithm "partb" in W. Riha, K. R. James (1976).

Remark

The original algorithm lead to BoundsErrors, since r could get smaller than 1. Furthermore x and y are handled as arrays with an infinite length. After finding all valid partitions, the algorithm will continue searching for partitions of length n+1. We thus had to add a few additional checks and interruptions. Done by T. Schmit.

ascending_partitions â Function
ascending_partitions(n::IntegerUnion;alg="ks")

Instead of encoding a partition of an integer $n â¥ 0$ as a descending sequence (which is our convention), one can also encode it as an ascending sequence. In the papers Kelleher & O'Sullivan (2014) and Merca (2012) it is said that generating the list of all ascending partitions is more efficient than generating descending ones. To test this, I have implemented the algorithms given in the papers:

  1. "ks" (default) is the algorithm "AccelAsc" (Algorithm 4.1) in J. Kelleher, B. O'Sullivan (2014).
  2. "m" is Algorithm 6 in M Merca (2012). This is actually similar to "ks".

The ascending partitions are stored here as arrays and are not of type Partition since the latter are descending by our convention. I am using "ks" as default since it looks slicker and I believe there is a tiny mistake in the publication of "m" (which I fixed).

Comparison

I don't see a significant speed difference to the descending encoding:

julia> @btime partitions(Int8(90));
3.376 s (56634200 allocations: 6.24 GiB)

julia> @btime ascending_partitions(Int8(90),alg="ks");
3.395 s (56634200 allocations: 6.24 GiB)

julia> @btime ascending_partitions(Int8(90),alg="m");
3.451 s (56634200 allocations: 6.24 GiB)

getindex_safe â Function
getindex_safe(P::Partition, i::IntegerUnion)

In algorithms involving partitions it is sometimes convenient to be able to access parts beyond the length of the partition, and then you want to get zero instead of an error. This function is a shortcut for

return (i>length(P.p) ? 0 : getindex(P.p,i))

If you are sure that P[i] exists, use getindex because this will be faster.
